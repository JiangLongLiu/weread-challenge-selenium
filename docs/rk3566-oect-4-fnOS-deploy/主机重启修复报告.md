# RK3566-OECT1-4-fnOS 主机重启后启动流程修复报告

## 修复时间
2026 年 4 月 22 日 23:09

## 问题背景

用户重启了 192.168.123.54 主机（RK3566-OECT1-4-fnOS），发现 Selenium 容器没有自动启动。

## 问题诊断

### 初始状态检查
- **系统启动时间**: 2026-04-22 22:59:17
- **systemd 服务状态**: `disabled; preset: enabled` ❌
- **容器状态**: 无运行中的容器 ❌
- **VNC 端口**: 未监听 ❌

### 问题根源

通过对比 192.168.123.52（RK3566-OECT1-2-fnOS）主机的配置，发现问题：

**54 号主机的问题：**
1. `/etc/init.d/weread-selenium` 存在（OpenWrt 格式）
2. systemd 尝试与 SysV 脚本同步
3. SysV 脚本使用 OpenWrt 格式（`#!/bin/sh /etc/rc.common`），fnOS 不支持
4. 导致 `systemctl enable` 失败，服务被禁用

**52 号主机的正确配置：**
1. `/etc/init.d/weread-selenium` 存在但**不使用**（仅作脚本文件）
2. `/etc/rc.d/S99weread-selenium` **不存在**
3. systemd 服务文件独立：`/etc/systemd/system/weread-selenium.service`
4. systemd 服务状态：`enabled` ✅

### fnOS 系统特点

- fnOS 基于 Debian，使用 **systemd** 作为初始化系统
- **不支持** OpenWrt 的 SysV 脚本格式（`#!/bin/sh /etc/rc.common`）
- 开机启动应使用纯 systemd 服务文件

## 修复步骤

### 1. 删除冲突的 SysV 脚本
```bash
rm -f /etc/init.d/weread-selenium
```
**原因**: 该脚本是 OpenWrt 格式，与 systemd 冲突

### 2. 清理 SysV 链接
```bash
rm -f /etc/rc.d/S99weread-selenium
```
**原因**: 防止 SysV 启动机制干扰

### 3. 重新加载 systemd 配置
```bash
systemctl daemon-reload
```

### 4. 启用 systemd 服务（关键步骤）
```bash
systemctl enable weread-selenium.service
```
**结果**: `enabled` ✅

### 5. 启动服务
```bash
systemctl start weread-selenium.service
```

## 关键发现

### fnOS 系统启动机制
1. **不使用** SysV 初始化脚本（`/etc/init.d/*`）
2. **使用** systemd 服务文件（`/etc/systemd/system/*`）
3. `systemctl enable` 创建 systemd 链接，不与 SysV 同步

### 为什么之前会失败
- 部署脚本创建了 OpenWrt 格式的 `/etc/init.d/weread-selenium`
- systemd 尝试与 SysV 脚本同步
- SysV 脚本没有正确的 Default-Start 配置
- 导致 `update-rc.d` 报错，`systemctl enable` 失败
- 服务保持 `disabled` 状态

### 正确的配置方式
1. **只使用** systemd 服务文件
2. **不使用** SysV 初始化脚本
3. **直接**使用 `systemctl enable/disable` 管理
