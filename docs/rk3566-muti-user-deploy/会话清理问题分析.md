# Selenium 会话清理问题分析与解决方案

## 问题起源

### 现象
用户手动启动 app-1 容器后，再手动停止容器并尝试清理 Selenium 会话，但 Selenium 中的浏览器进程未关闭：

```bash
# 停止容器并调用 DELETE API
docker compose stop app-1 && curl -s -X DELETE http://127.0.0.1:4444/se/grid/session

# 返回错误
{
  "value": {
    "message": "Unable to find handler for (DELETE) /se/grid/session",
    "error": "unknown command"
  }
}
```

### 问题分析

1. **Selenium Grid 4.x API 变更**
   - Selenium 4.x 使用 WebDriver BiDi 协议
   - 旧的 Grid API 路径 `/se/grid/session` 已不存在
   - 正确的会话删除 API 应该是 `DELETE /session/{sessionId}`，但需要具体的会话 ID

2. **容器停止不等于会话清理**
   - 停止 app 容器只会断开与 Selenium 的连接
   - Selenium 中的浏览器进程（Chrome）仍在运行
   - 长时间累积会导致内存泄漏

## 解决方案

### 方案对比

| 方案 | 实现方式 | 效果 | 问题 |
|------|---------|------|------|
| curl DELETE API | 调用 `/se/grid/session` | ❌ 无效 | API 路径不存在于 Selenium 4.x |
| 获取 sessionId 后删除 | 先 GET `/sessions` 再 DELETE `/session/{id}` | ⚠️ 复杂 | Selenium 4.x 的 `/sessions` 端点也已变更 |
| **docker restart** | 重启 Selenium 容器 | ✅ **有效** | 简单粗暴，彻底清理所有进程 |

### 采用方案：docker restart

**原理**：
- 重启 Selenium 容器会终止容器内所有进程
- Chrome 浏览器进程被强制关闭
- 容器重新启动后，环境干净

**验证结果**：
```bash
# 重启前 Chrome 进程数
$ docker exec weread-challenge-selenium-muti-user-selenium-1 ps aux | grep chrome | wc -l
7

# 重启后 Chrome 进程数
$ docker restart weread-challenge-selenium-muti-user-selenium-1
$ docker exec weread-challenge-selenium-muti-user-selenium-1 ps aux | grep chrome | wc -l
1
```

## 实施细节

### 1. 更新定时任务

将原来的 curl DELETE 命令替换为 docker restart：

```bash
# 旧命令（无效）
curl -s -X DELETE http://127.0.0.1:4444/se/grid/session 2>/dev/null || true

# 新命令（有效）
docker restart weread-challenge-selenium-muti-user-selenium-1
```

### 2. 完整任务流程

每个定时任务现在执行：

```bash
# 1. 启动用户容器
docker compose up app-X -d

# 2. 等待 70 分钟（4200 秒）
sleep 4200

# 3. 停止用户容器
docker compose stop app-X

# 4. 重启 Selenium 清理会话
docker restart weread-challenge-selenium-muti-user-selenium-1
```

### 3. 部署脚本更新

`ssh_scp_util.py` 中的 `setup_cron` 函数已更新，所有 16 个定时任务都使用新的清理方式。

## 经验总结

1. **API 文档要对应版本**：Selenium 4.x 与 3.x 的 API 差异很大，不能混用
2. **简单方案往往更有效**：docker restart 比复杂的 API 调用更可靠
3. **验证必不可少**：实际测试才能发现问题，不能只看命令是否执行成功
